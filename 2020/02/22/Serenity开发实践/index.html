应用服务、领域服务和数据交互对象

应用服务（Application Service）是暴露给前端或者其他系统调用的方法。应用服务之间最好不要互相调用；权限应定义在应用服务层之上。

领域服务(Domain Service)是用来实现业务逻辑，所以领域服务的代码量最多，也最重要。写代码者一定要完全理解业务流程，并能够抽象出业务流程的生命周期。

数据交互对象(Data Transfer Object)分为两种，一种是对外交互的数据，比如前端或者其他系统调用应用服务传递的数据(Application Service DTO)；一种是领域服务间互相调用或者应用服务调用领域服务时传递的数据对象(Domain Service DTO)。每一个应用服务都需要定义DTO, 不同的应用服务可以共享一个DTO. 领域服务可以对一些通用的对象，抽象成为DTO。

为每一个流程建一个应用服务,领域服务和数据交互对象

管理系统大部分都是由一个一个流程组成，为每一个流程建立一个文件夹，包含其所需的应用服务，领域服务和数据交互对象，这样代码结构清晰，方便理解和查找。下图中包含4个流程，4M变化点，生产任务执行，产线问题处理和QRAP流程。后缀为Dto的是数据交互对象，后缀为Repository是领域服务，后缀为Service的是应用服务。在有些流程，因为定义了较多的领域服务Dto, 所以我也将其专门放一个文件。

如何设计领域服务

再次强调，领域服务的设计非常重要。优秀的领域服务设计有下面两个特点：

  1. 代码是复用的，一个业务逻辑必须在一处代码中实现；复用的代码可以使你的程序容易维护。
  2. 逻辑是方便扩展的，管理系统的逻辑变更几乎是不可避免的，优秀的程序设计将只需要更改少量代码，即可实现添加或修改系统功能，而一个糟糕的程序设计，将使得变更几乎不可能；

遵循以下几点可以设计出合理的领域服务：

  1. 充分理解业务流程，绘制流程图

  2. 抽象出业务流程的生命周期，以生产问题处理流程为例，其生命周期可以为

    • 创建：产线工人填写问题
    • 提交：班组长查看、过滤
    • 通知：传递给对应的部门去处理
    • 解决：对应的部门解决问题
    • 确认：解决完之后由上报的人确认
    • 关闭：关闭问题

  3. 为每一个针对Entity的操作写一个独立的方法，可以使代码尽量复用。

    如果某几个针对Entity 的操作总是一起出现，逻辑上不可分离，应写在一个方法内，以免使方法太多，识别这些逻辑不可分离的Entity操作非常重要，通过绘制流程图可以帮助我们方便识别这些操作

  4. 方法的命名应该尽量简洁，比如Create, Submit, Cancel_Create, Cancel_Submit. 如果需要操作其他Entity, 可以命名为{action}{entity}, cancel{action}_{entity}

  5. 将方法按照生命周期排放，并用#Region包裹起来

  1. 抽象一些通用的领域DTO

    抽象一些通用的领域DTO是必要的，否则有些领域服务的方法将需要传递非常多的参数，不容易编码也不容易扩展。比如提交质检结果，其内容包含有合格数量、报废数量、回用数量、返工数量、完工数量、移交数量、质检员、检验时间等8个属性，而且这8个属性总是同时被需要，这时我们可以将其设计为一个DTO，并且该DTO可以包含自己的逻辑。

public class InspectionResultDto {
        public Decimal? PassQty = 0;
        public Decimal? ScrapQty = 0;
        public Decimal? ReUseQty = 0;
        public Decimal? ReWorkQty = 0;
        public Datetime InpectionTime;
        public Int32 InspectorUserId;
        public Decimal TotalQty
        {
            get
            {
                return (Decimal)PassQty + (Decimal)ScrapQty + (Decimal)ReUseQty + (Decimal)ReWorkQty;
            }
        }
        public Decimal HandOverQty {
            get {
                return (Decimal)PassQty + (Decimal)ReUseQty;
            }
        }

    }

  1. 领域服务设计图

    领域服务包含的代码最多，一个领域服务可能包含几十个方法。领域服务方法之间存在互相调用，逻辑复杂。所以通常需要几次的检查、重构，来保持设计的正确性。

    以下是领域服务的设计图，紫色方框代表一个实现的具体业务，其是一组基础领域方法的组合。比如生产订单的释放，包含一组动作，

    - 检查是否满足释放条件
- 创建释放记录
- 反改生产订单释放数量
- 创建该释放记录对应的工序计划任务
- 查询首到工序
- 判断工序是否自动释放
- 释放工序（释放工序也是一个具体业务，包含几个基础领域方法）

    其中创建释放记录，反改生产订单释放数量，创建对应的工序计划任务总是一起出现，不可分割，可以写成一个方法。

    通过不断的对该图进行Review、调整来优化我们的领域服务设计。
